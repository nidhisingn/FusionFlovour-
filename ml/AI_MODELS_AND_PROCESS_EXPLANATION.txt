AI / ML EXPLANATION FOR THE RECIPE PREDICTION PROJECT
=====================================================

1. PROJECT GOAL
---------------
The goal of this AI module is to predict a suitable recipe from the ingredients entered by the user.

Example:
- Input: tomato, onion, garlic, rice
- Output: a recipe suggestion such as fried rice, soup, curry, or another matching dish.

This is mainly a machine learning based recommendation and classification system.


2. WHAT PROBLEM THE AI SOLVES
-----------------------------
The AI layer helps answer this question:

"Given a set of ingredients, what dish or recipe is the best match?"

Earlier, the project tried to directly predict an exact recipe title. That approach became weak because the dataset had very few rows and almost every recipe title appeared only once.

So the improved system now does two things:

1) It predicts a broader recipe family/group first
   - examples: soup, curry, salad, rice, breakfast, dip/spread, wrap/roll

2) It then re-ranks real recipes using ingredient overlap and metadata match
   - this makes the final answer more practical and more stable.


3. OVERALL AI PIPELINE
----------------------
The complete AI process in this project is:

Step 1: Data collection
- Data is fetched from Spoonacular using API scripts.
- Important fields collected:
  - recipe title
  - ingredients
  - cuisines
  - diets
  - dishTypes

Step 2: Data preparation
- Ingredients are cleaned and normalized.
- Pipe-separated ingredient strings are converted into lists.
- Metadata such as cuisine, diet, and dish type are also converted into usable lists.

Step 3: Target engineering
- Instead of directly learning every exact recipe title, recipes are grouped into broader categories.
- Example groups:
  - soup
  - salad
  - curry
  - rice
  - pasta_noodles
  - dip_spread
  - breakfast
  - wrap_roll
  - side_dish
  - misc_recipe

Step 4: Feature engineering
- Ingredient text is converted into TF-IDF features.
- Combined text is created using:
  - ingredients
  - cuisine metadata
  - diet metadata
  - dish type metadata
  - recipe title tokens
- Metadata is also encoded separately as sparse binary features.

Step 5: Model training
- Multiple ML models are trained and compared.
- Cross-validation is used instead of only one train/validation split.

Step 6: Model selection
- The best model is selected based mainly on top-3 performance.
- Top-3 is important because recipe prediction is often recommendation-style, where multiple near matches are useful.

Step 7: Retrieval / reranking
- The model predicts recipe-group probabilities.
- Then the system compares user ingredients against the real recipes stored in the retrieval index.
- Final ranking uses:
  - model confidence
  - ingredient overlap score
  - metadata match score

Step 8: Final enriched response
- The project returns:
  - predicted recipe
  - predicted group
  - alternatives
  - substitutions
  - diet compatibility
  - cuisine hints
  - personalization hint


4. WHY THE OLD APPROACH NEEDED IMPROVEMENT
------------------------------------------
Earlier issues in the ML system were:

1) Very small dataset
- The previous metrics showed around 34 rows and 34 classes.
- That means nearly one sample per class.
- This is too little for exact-recipe classification.

2) Direct exact-label prediction
- Predicting exact recipe titles with tiny data causes poor generalization.

3) Single split evaluation
- Using one train/validation split on tiny data is unstable.

4) Limited feature representation
- Earlier the model mainly relied on ingredient multi-hot vectors.
- That is useful, but it does not fully use metadata and textual relationships.

5) Weak ranking alone
- Even if the classifier is weak, recipe search can still be strong if retrieval and overlap scoring are used.


5. IMPROVEMENTS IMPLEMENTED IN THIS PROJECT
-------------------------------------------
The following improvements were added:

1) Richer features using TF-IDF
- Ingredients are now converted into TF-IDF unigram and bigram features.
- Why this helps:
  - captures ingredient importance better than only binary presence
  - supports phrase patterns and co-occurrence better

2) Metadata-based features
- cuisines, diets, and dishTypes are added as model features.
- Why this helps:
  - the same ingredients can belong to different recipe styles
  - metadata gives useful context

3) Grouped target labels
- Exact recipe title prediction was replaced by broader grouped recipe families.
- Why this helps:
  - reduces class sparsity
  - makes the model more learnable
  - improves practical prediction quality

4) Cross-validation
- Stratified K-Fold cross-validation is used.
- Why this helps:
  - gives more reliable evaluation on small datasets
  - reduces dependency on a single lucky or unlucky split

5) Hybrid reranking / retrieval
- Model output is combined with recipe overlap scoring.
- Why this helps:
  - makes predictions more user-relevant
  - gives better real-world ranking than pure classification alone

6) Metadata-aware ranking
- Preferred cuisines and diet preferences influence the final ranking.
- Why this helps:
  - gives more personalized suggestions

7) Low-frequency group stabilization
- Very small classes are lightly augmented in training.
- Why this helps:
  - avoids CV/training instability in demo conditions
- Important note:
  - this is only a temporary improvement for prototype quality
  - real dataset growth is still the best solution


6. MODELS USED IN THE PROJECT
-----------------------------
The project trains multiple models so that we can compare different learning behaviors.


6.1 Logistic Regression (One-vs-Rest)
-------------------------------------
What it is:
- A linear classification model.
- In multiclass settings here, it is used through One-vs-Rest.

Why we used it:
- It is a strong baseline for sparse text-like features.
- TF-IDF features often work very well with linear models.
- It is fast, stable, and interpretable.

Purpose in this project:
- To act as a reliable benchmark model.
- To check whether simple linear separation is enough for recipe-group prediction.

Strengths:
- fast training
- strong on sparse high-dimensional features
- easy to compare

Weaknesses:
- may miss complex non-linear relationships


6.2 Linear SVC with Calibration (One-vs-Rest)
---------------------------------------------
What it is:
- A Support Vector Machine based linear classifier.
- Calibration is used so it can provide probability-like outputs.

Why we used it:
- SVM-style linear classifiers are very strong on sparse text features.
- Recipe ingredient data behaves similarly to text classification problems.

Purpose in this project:
- To achieve stronger separation between recipe groups.
- To produce good ranking scores for top-k suggestions.

Strengths:
- usually very strong on sparse NLP-like features
- good margin-based classification
- often better ranking than basic baselines

Weaknesses:
- less interpretable than logistic regression
- calibration adds extra training complexity


6.3 Multinomial Naive Bayes
---------------------------
What it is:
- A probabilistic model often used for text classification and token-count problems.

Why we used it:
- Ingredients represented with TF-IDF or tokenized text behave somewhat like document features.
- Naive Bayes is lightweight and often surprisingly strong on sparse inputs.

Purpose in this project:
- To provide a simple and fast probabilistic baseline.
- To compare against more complex linear and neural models.

Strengths:
- very fast
- simple
- works well on sparse feature spaces

Weaknesses:
- assumes independence between features
- may oversimplify ingredient interactions


6.4 Random Forest
-----------------
What it is:
- An ensemble of decision trees.

Why we used it:
- To test whether non-linear ingredient combinations can be captured better than linear models.

Purpose in this project:
- To model interactions between features that linear models may not capture.

Strengths:
- handles non-linear relationships
- robust to noise
- captures feature interactions

Weaknesses:
- often not ideal for extremely sparse text-style data
- can become heavy and less effective than linear models for TF-IDF features


6.5 sklearn MLP (Neural Network)
--------------------------------
What it is:
- A feed-forward neural network using `MLPClassifier`.

Why we used it:
- To check whether a non-linear neural model can learn better patterns from the richer features.

Purpose in this project:
- To explore whether deeper feature interactions improve grouped recipe prediction.

Strengths:
- can model non-linear patterns
- flexible compared to linear models

Weaknesses:
- needs more data than linear models to shine
- can overfit on small datasets


7. WHY MULTIPLE MODELS ARE TRAINED
----------------------------------
We train multiple models because no single algorithm is always best.

Reasons:
- Different models behave differently on sparse, small, and metadata-rich datasets.
- It helps compare linear vs probabilistic vs tree-based vs neural behavior.
- It improves confidence in the final system design.
- The project requirement also asked for multiple models.


8. WHY GROUPED PREDICTION IS BETTER THAN EXACT RECIPE PREDICTION HERE
---------------------------------------------------------------------
In this dataset, predicting exact recipe names is difficult because:
- many recipe titles appear only once
- classes are too sparse
- small data causes unstable exact-label learning

Grouped prediction is better because:
- the model learns broader food patterns
- evaluation becomes more meaningful
- retrieval can handle the last-mile exact recipe selection

So the new architecture is:

Ingredients -> Predict recipe group -> Re-rank real recipes -> Return best recipe

This is a more practical hybrid AI design.


9. FEATURE ENGINEERING DETAILS
------------------------------

9.1 Ingredient text
- All ingredient lists are joined as text.
- Example:
  tomato onion garlic rice

9.2 TF-IDF features
- TF-IDF converts tokens into weighted numeric vectors.
- Common ingredients receive lower weight.
- More informative ingredients receive higher weight.

9.3 Unigrams and bigrams
- The model uses unigram + bigram features.
- This helps represent single ingredients and ingredient combinations.

9.4 Metadata encoding
- cuisine, diet, and dish type are added as sparse metadata features.
- Example:
  cuisine_indian
  diet_vegan
  dish_soup

9.5 Combined text feature
- Ingredients and metadata are also combined into one richer text field.
- This helps models learn relations between ingredients and style cues.


10. EVALUATION STRATEGY
-----------------------
The improved pipeline uses cross-validation.

Why:
- A single split can be misleading on small data.
- Cross-validation tests the model on multiple folds.

Metrics considered:
- Accuracy
- Top-3 accuracy

Why top-3 matters:
- In recipe recommendation, there may be several valid outputs.
- If the correct or relevant group is in the top 3, the system is still useful.


11. RETRIEVAL AND RERANKING PROCESS
-----------------------------------
After the classifier predicts probabilities for grouped classes, the project does not stop there.

It also builds a retrieval index of real recipes.

Then, for each recipe, it computes:

1) Model score
- how likely the model thinks that recipe’s group is

2) Ingredient overlap score
- exact overlap
- partial overlap
- coverage and precision signals

3) Metadata match score
- preferred cuisines
- diet preference

Final score combines these signals.

This hybrid approach is very important because:
- users care about actual recipes, not just class labels
- overlap-based reranking increases practical quality


12. PERSONALIZATION AND ENHANCEMENT LAYER
-----------------------------------------
Besides core ML prediction, the system also includes:

1) Ingredient normalization
- maps common ingredient synonyms into normalized names
- example: capsicum -> bell pepper

2) Substitution suggestions
- suggests alternatives for common ingredients

3) Diet compatibility checking
- checks whether ingredients conflict with a selected diet

4) Cuisine hinting
- infers rough cuisine tags from recipe names

5) History-based personalization hint
- gives a light user-oriented explanation based on previous predictions

These features improve user experience even when the classifier is not perfect.


13. IMPORTANT LIMITATION STILL REMAINING
----------------------------------------
Even after these improvements, the biggest limitation is still the dataset size.

The best future improvement is:
- collect many more real recipes
- especially multiple examples per recipe group
- replace synthetic augmentation with actual data

This will improve all models much more than only changing the algorithm again.


14. BEST FUTURE IMPROVEMENTS
----------------------------
Recommended next improvements:

1) Increase dataset size significantly
2) Collect more balanced samples per recipe group
3) Add better ingredient normalization and ontology mapping
4) Use embeddings / transformer-based ingredient representations later
5) Add user feedback loop to learn from accepted suggestions
6) Store prediction outcomes and use them for re-ranking improvement
7) Add stronger hyperparameter search
8) Introduce separate offline evaluation datasets
9) Build a real retrieval engine with vector search if the dataset grows large


15. FINAL CONCLUSION
--------------------
This project now uses a more realistic AI architecture for recipe suggestion:

- multi-model training
- richer TF-IDF and metadata features
- grouped class prediction
- cross-validation evaluation
- retrieval-based reranking
- user-oriented explanation and personalization support

Why this is better:
- It is more stable than direct exact recipe classification on tiny data.
- It gives more meaningful predictions.
- It is easier to improve in the future as more data is collected.

In simple terms:
The project does not rely on one model alone anymore. It now combines classification, metadata understanding, and recipe retrieval logic to give better recipe suggestions from ingredients.
